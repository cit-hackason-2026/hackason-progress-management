# SEMP（Systems Engineering Management Plan）

システムズエンジニアリングの観点から、プロジェクト全体をどのように計画・実行・検証するかを定義する。

## 1. 目的・スコープ
- 本プロジェクトにおけるSE活動の範囲と役割を明確にする
- V字モデルに沿って、要件定義〜受入までを一貫して管理する

## 2. 開発プロセス（V字モデル）
```
要件定義(MOE)  ─────────────────── 受入テスト(MOE検証)
   │                                    │
   基本設計(PBS)  ──────────── システムテスト(MOP検証)
        │                          │
        詳細設計  ─────── 結合テスト(MOP検証)
             │            │
             実装 ── 単体テスト(MOP検証)
```

## 3. 成果物とフェーズの対応
| フェーズ | 主な成果物 | 参照 |
|---------|-----------|------|
| 要件定義 | 要件定義書、FBS、MOE | [FBS](../02-scope/fbs.md) / [MOE](../05-quality/moe.md) |
| 基本設計 | アーキテクチャ図、PBS、MOP | [PBS](../02-scope/pbs.md) / [MOP](../05-quality/mop.md) |
| 詳細設計・実装 | 設計書、ソースコード、TPM | [TPM](../05-quality/tpm.md) |
| 検証 | テスト仕様・結果、V&V報告 | [V&V](../05-quality/vandv.md) |
| 受入 | 受入テスト結果、デモ | [MOE](../05-quality/moe.md) |

## 4. 技術成熟度の目標
- プロジェクト開始時 TRL1 → 最終発表時 TRL3〜4 を目指す
- 詳細は [TRL](../05-quality/trl.md) を参照

## 5. 役割分担（SE観点）
| 役割 | 責任 |
|------|------|
| システムエンジニア | 要件〜V&Vの整合性確保 |
| PM | WBSによる作業管理 |
| 開発メンバー | 各成果物の作成・テスト |

## 6. 技術的リスクの監視
- TPMを用いて開発中に継続監視
- 閾値を下回った場合は [リスク登録簿](../08-risk/risk-register.md) に起票

## 7. 変更管理
- 要件・設計の変更は [decision-log](./decision-log.md) に記録し、関連する FBS/PBS/MOE/MOP を同期する
